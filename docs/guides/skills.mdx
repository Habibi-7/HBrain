## Overview

The default second-brain skill captures notes, tasks, decisions, facts, and links. Custom skills let you define a completely different capture workflow for any domain — meeting notes, research tracking, project logs, bug reports, reading journals, whatever fits how you work.

Each custom skill is a directory at `~/brain/skills/<name>/` containing:
- `SKILL.md` — the skill definition your agent reads
- `vault/events/` — isolated event storage for that skill

## Creating a skill

```bash
brain skill create <name> [description]
```

Examples:

```bash
brain skill create "Meeting Notes" "Track meetings and action items"
brain skill create "Research Tracker" "Papers, citations, and reading notes"
brain skill create "Bug Log" "Capture bugs, root causes, and fixes"
```

Names are normalized to lowercase with hyphens (`Meeting Notes` → `meeting-notes`).

After creation, the CLI prints the path to the generated `SKILL.md`:

```
✓ Skill created: meeting-notes
  ~/brain/skills/meeting-notes/SKILL.md

  Edit SKILL.md to customize event types, triggers, and capture rules.
```

## Listing and inspecting skills

```bash
brain skill list              # all skills with descriptions and event types
brain skill show <name>       # full details: version, vault path, triggers
brain skill path <name>       # print path to SKILL.md (useful for scripting)
```

## Anatomy of a SKILL.md

Every generated skill has this frontmatter:

```yaml
---
name: meeting-notes
description: |
  Track meetings and action items
version: 1.0.0
event_types: [note, task, decision, fact, link]
task_statuses: [open, done, blocked, cancelled]
vault_path: ~/brain/skills/meeting-notes/vault
triggers:
  - meeting-notes
  - /meeting-notes
---
```

The body contains sections for vault setup, capturing events, querying, and output style. These are instructions your agent reads and follows.

## Customizing your skill

Open the SKILL.md in any editor — it's plain markdown:

```bash
# Open in your default editor
open "$(brain skill path meeting-notes)"

# Or with a specific editor
code "$(brain skill path meeting-notes)"
```

### Change event types

Replace the defaults with types that fit your domain:

```yaml
event_types: [meeting, action-item, decision, follow-up]
```

Then update the event types table in the body to describe when to use each one:

```markdown
### Event types

| Type | When |
| --- | --- |
| `meeting` | Log for each meeting: attendees, agenda, notes |
| `action-item` | Something someone committed to doing |
| `decision` | A choice made during the meeting |
| `follow-up` | Something to revisit at the next meeting |
```

### Change task statuses

```yaml
task_statuses: [open, done, delegated, cancelled]
```

### Add trigger phrases

Add phrases that activate capture for this skill:

```yaml
triggers:
  - meeting notes
  - we had a meeting
  - sync notes
  - standup notes
  - action item
  - /meeting
```

### Write domain-specific capture rules

The capture rules section tells your agent how to behave. Be specific:

```markdown
### Capture rules

1. Always record who was in the meeting (attendees) in the body.
2. Action items always go as `action-item` type with `status: open` and a tag for the owner.
3. Use the meeting date as `created_at`, not the time of capture.
4. Keep decisions short — the reasoning, not the discussion.
```

### Change the vault path

By default, each skill stores events in its own subdirectory. You can point it anywhere:

```yaml
vault_path: ~/meetings/vault
```

Update the vault setup section and all file paths in the body to match.

## Loading a skill into your agent

A custom skill is activated the same way as the default brain skill — load the `SKILL.md` into your agent.

**Claude Code / Cowork:**

```bash
# Copy to your skills directory
cp ~/brain/skills/meeting-notes/SKILL.md ~/.claude/skills/meeting-notes.md

# Or symlink it so edits stay in sync
ln -s ~/brain/skills/meeting-notes/SKILL.md ~/.claude/skills/meeting-notes.md
```

**Other agents:** Load the SKILL.md content wherever your agent platform reads skill files.

Once loaded, the agent responds to the triggers you defined and writes events to that skill's vault path.

## Running multiple skills

You can have multiple skills active at once. Each agent session can load whichever skills apply to the work at hand.

```bash
brain skill list
```

```
Skills (3)

  second-brain         Capture personal notes, tasks, decisions
                       types: note, task, decision, fact, link

  meeting-notes        Track meetings and action items
                       types: meeting, action-item, decision, follow-up

  research-tracker     Papers, citations, and reading notes
                       types: paper, citation, note, question
```

Each skill has its own isolated vault, so events never mix between skills.

## Versioning skills

The `version` field in frontmatter lets you track changes:

```yaml
version: 1.2.0
```

Bump it when you change event types or capture rules significantly — this helps if you share skills with a team and need to coordinate updates.

## Sharing skills

A skill is just a markdown file. Share it by:
- Committing it to a git repo
- Sending the SKILL.md file directly
- Publishing it anywhere markdown renders

The recipient drops it into their `~/brain/skills/<name>/` directory (create the directory if it doesn't exist) and loads it into their agent.
